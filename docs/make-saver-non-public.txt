Goal
----
Remove the /saver/ location block from nginx/nginx.conf.template so that the
saver service is no longer reachable from the public internet on cyber-dojo.org.

Why saver is currently public
------------------------------
nginx is the only internet-facing service (the only one registered as an ALB
target). It proxies /saver/ to the saver service:

  location /saver/ {
    rewrite ^/saver/(.*) /$1 break;
    proxy_pass http://$saver_hostname:${CYBER_DOJO_SAVER_PORT};
  }

This means any caller who knows a kata ID can invoke saver API methods directly
(kata_events, kata_ran_tests, kata_option_set, kata_download, etc.) without going
through the web service. The only protection is that kata IDs are unguessable.

Why the nginx proxy cannot simply be deleted yet
-------------------------------------------------
Several pieces of browser-side JavaScript in the web service make AJAX calls
directly to /saver/... paths (i.e. the browser calls saver via nginx, bypassing
the web backend). These would all break if the nginx proxy were removed:

  web/source/app/assets/javascripts/cyber-dojo_lib.js:12
    $.getJSON(`/saver/${name}`, ...)          -- kata_events (and others, dynamic)

  web/source/app/assets/javascripts/cyber-dojo_kata_manifest.js:11
    url: `/saver/kata_manifest?id=${id}`

  web/source/app/views/shared/_settings.erb:78,95
    url: '/saver/kata_option_get'
    url: '/saver/kata_option_set'

  web/source/app/views/kata/_download.erb:27
    url: '/saver/kata_download'

  web/source/app/views/review/_fork_button.erb:72
    url: `/saver/${method}?id=...`            -- kata_fork, group_fork

  web/source/app/views/review/_output.erb:13
    review.getJSON('saver', 'kata_event', ...)

  web/source/app/views/review/_avatar_navigator.erb:25
    review.getJSON('saver', 'group_joined', ...)

  web/source/app/assets/javascripts/cyber-dojo_lib_open_avatar_selector_dialog.js:27
    cd.review.getJSON('saver', 'group_joined', ...)

Required refactor
-----------------
Each of the above JavaScript calls needs to be rerouted through the web
controller instead of hitting /saver/ directly. The web controller already calls
saver server-side on the internal Docker/VPC network; the same pattern should be
extended to cover these endpoints. The browser would call a web route (e.g.
/kata/events, /kata/manifest, etc.) and the web controller would proxy the
request to saver internally.

Once all browser-side /saver/ calls are replaced, the /saver/ location block can
be deleted from nginx/nginx.conf.template.

Effect on non-production environments
--------------------------------------
In both 'cyber-dojo up' (self-hosted) and 'make demo' (developer), saver has no
published Docker ports -- only nginx is port-mapped to the host. Saver is already
unreachable from outside Docker in these environments. Removing the nginx proxy
location would have no effect on network accessibility there; web reaches saver
directly on the cyber-dojo bridge network regardless.

The change only has a security effect in the production (cyber-dojo.org)
deployment, where nginx is behind the public ALB.

/runner/ is already safe
-------------------------
The runner's /run endpoint (which executes code) is not exposed through nginx at
all. Only /runner/sha and /runner/base_image are proxied, and no browser-side
JavaScript calls either of those paths -- they can be removed from nginx without
any refactoring.

What this refactoring does and does not achieve
------------------------------------------------
After all /saver/ browser calls are moved to web controller routes, the refactoring:

  DOES:
  - Remove direct external access to the saver service's internal API.
    Someone can no longer curl /saver/kata_events (as documented in the
    saver repo's API docs) to reach saver methods directly.
  - Put all kata data access behind the web controller, which is the right
    place to add authentication or rate-limiting in the future.

  DOES NOT:
  - Prevent a determined person from curling the new web controller routes
    (e.g. /kata/events?id=XXXX) to read kata data without a browser. The
    web controller routes are plain public HTTP endpoints forwarded by
    nginx's catch-all location / block.
  - Address creator/, dashboard/, or differ/ -- these services are also
    proxied directly by nginx and are reachable via curl in the same way
    saver was.

CSRF vs authentication
----------------------
The web app has a CSRF token mechanism. In app.rb the enforcement check is
currently commented out:

  # Phase 2: uncomment once all users have reloaded and have the csrf_token cookie
  # unless %w[GET HEAD OPTIONS TRACE].include?(request.request_method)
  #   halt 403, 'Forbidden' unless params['authenticity_token'] == @csrf_token
  # end

CSRF tokens protect against cross-site request forgery -- a malicious
third-party website tricking a user's browser into making requests on their
behalf. They do NOT prevent deliberate curl access: a curl user can read the
CSRF cookie from a real browser session and include it in their request.

To genuinely prevent unauthenticated curl access to kata data, proper
authentication (session tokens, API keys, etc.) would be needed on the web
controller routes. That is a separate piece of work beyond this refactoring.
