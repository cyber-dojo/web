Rate limiting in cyber-dojo
============================


Trace: what happens when [test] is pressed
-------------------------------------------

1. Browser POSTs to /kata/run_tests/:id
2. nginx -- applies limit_req here; rejects with 429 if over rate, otherwise proxies to web
3. web -- runs tests via runner service, collects results, calls saver.kata_ran_tests(...)
4. saver -- receives the POST directly from web (Docker network)

Rate limiting at step 2 protects both runner and saver indirectly: if nginx drops
the browser request before it reaches web, web never makes either downstream call.
Neither runner nor saver is directly exposed to the rate limiter.


Why runner and saver cannot be rate-limited independently for run_tests
-----------------------------------------------------------------------

Runner and saver are coupled in the same web request handler: web calls runner,
waits for the result, then calls saver.kata_ran_tests. There is no path where
saver receives a kata_ran_tests call without runner having already been called.
Separating them would require decoupling the flow (e.g. an async queue or a
separate browser-initiated save call), which is a significant architectural change.

The meaningful separation is that saver has POST endpoints that do not involve
runner at all: kata_file_edit, kata_file_create, kata_file_delete, kata_file_rename.
Those go web -> saver with no runner in the path and could be rate-limited
independently if needed.


Why rate limiting at the nginx->web boundary is preferred over routing
web->saver through nginx
----------------------------------------------------------------------

An alternative design would route web's internal saver calls through nginx
(web -> nginx -> saver) so that nginx could apply limit_req to the saver leg.
The problem is that nginx would see web's Docker IP as the remote address, not
the browser's IP. Rate limiting by $binary_remote_addr would throttle the whole
web service as a single client. Keying on $http_x_forwarded_for (the forwarded
browser IP) would work but requires web to propagate that header on internal
calls, which is fragile. Rate limiting at the browser->nginx boundary is simpler
and uses the real client IP directly.


Changes to nginx.conf.template
--------------------------------

Add the limit_req_zone directive outside the server block (at http context):

    limit_req_zone $uri zone=run_tests:1m rate=1r/10s;

Add a specific location block for run_tests BEFORE the catch-all location /:

    location ~ ^/kata/run_tests/ {
      limit_req zone=run_tests nodelay;
      limit_req_status 429;

      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
      proxy_set_header X-Forwarded-Proto $scheme;
      proxy_set_header Host $http_host;
      proxy_redirect off;
      proxy_pass http://$web_hostname:${CYBER_DOJO_WEB_PORT};
    }


Design decisions
-----------------

$uri as the zone key
  The rate limit is per kata ID (the full URI is /kata/run_tests/<id>),
  so two people working on different katas are unaffected by each other.
  Switching to "$uri$binary_remote_addr" would give per-kata-per-IP limits,
  which avoids two collaborators on the same kata sharing the same quota.

1r/10s rate
  One submission per 10 seconds per kata. A human would never hit this.
  Tune tighter if needed (e.g. 6r/m).

nodelay
  Requests over the limit are rejected immediately with 429 rather than
  queued. No benefit in buffering automated calls.

1m zone size
  1 MB holds approximately 16,000 tracked URIs, enough for a busy server.

Placement of the location block
  The specific location ~ ^/kata/run_tests/ block must appear before the
  catch-all location / block so nginx matches it first.


Limitations
------------

This does not distinguish AI from human -- it limits rate regardless of
origin. That is intentional: the rate is the problem, not the caller.

A sophisticated attacker spreading calls across many kata IDs would bypass
per-kata rate limiting. At that point the problem becomes runner resource
exhaustion, which is a separate concern (runner sandboxing, timeouts).


Rate limiting kata creation
============================

Goal: prevent automated clients from bulk-creating group or individual katas.

The plan is to rate-limit, by IP address, the endpoints in the saver service
that create group katas and individual katas. As with run_tests, this is best
done at the nginx->web boundary using $binary_remote_addr as the zone key,
so nginx sees the real browser IP.
